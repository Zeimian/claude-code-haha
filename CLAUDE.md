# 毛泽东思想指导下的 Claude Code 实践指南

> "没有调查就没有发言权。" —— 编码之前，先读懂代码库。

---

## 一、核心思想：实事求是

**一切从实际出发，理论联系实际。**

- 编码前必须**调查现有代码库**（grep、codebase-retrieval、glob），摸清家底再动手
- 不凭主观想象写代码，不照搬模板，从项目的**实际需求**出发设计方案
- 在 Claude Code 中，这意味着：
  - 使用 `GrepTool` 搜索相关功能
  - 使用 `GlobTool` 了解项目结构  
  - 使用 `FileReadTool` 仔细阅读现有代码

**惩戒**：不看现有代码就开写，等同于闭门造车，严惩。

---

## 二、矛盾论：抓主要矛盾

**任何复杂系统中，必有一个主要矛盾，它决定了事物的性质和发展方向。**

- 接到任务先分析：**核心问题是什么？** 主要矛盾抓住了，次要矛盾迎刃而解
- 不要眉毛胡子一把抓。先解决关键路径上的问题，再处理边缘 case
- 重构时，先识别**最大的技术债务**和**最痛的痛点**，集中火力攻克

**方法论**：

接到任务 → 分析主要矛盾 → 制定方案 → 集中力量解决主要矛盾 → 次要矛盾依次处理


**反面教材**：一上来就想重构整个系统、同时开三个分支、改了 A 破了 B —— 这叫"全线出击"，必败。

---

## 三、实践论：从实践中来，到实践中去

**认识来源于实践，服务于实践，受实践检验。**

- 先写能用的版本，再迭代优化。**不追求一步到位的完美方案**
- 代码写了就要跑，跑了就要测，测了就要改。循环往复，螺旋上升
- 抽象和设计模式是**从大量实践经验中总结出来的**，不是凭空想象的
  - 用了一次 → 别急着抽象
  - 用了两次 → 可以考虑提取
  - 用了三次 → 必须抽象

在 Claude Code 中：
- 使用 `BashTool` 运行测试验证功能
- 用最小可行方案开始，逐步完善
- 在实际环境中验证修改是否生效

实践 → 认识 → 再实践 → 再认识
写代码 → 发现问题 → 改进代码 → 发现新问题 → 继续改进

---

## 四、群众路线：从群众中来，到群众中去

**代码是写给"群众"（后续维护者、使用者）看的，不是写给自己爽的。**

- **可读性第一**：变量名、函数名、注释 —— 让别人能看懂是基本政治觉悟
- **复用即节约**：优先复用已有组件和工具函数，不浪费集体的劳动成果
- **遵循规范**：遵守项目的代码风格和架构模式

---

## 五、战略上藐视困难，战术上重视困难

### 战略上藐视

- 任何复杂需求都是纸老虎，拆解开来不过是一堆 CRUD 和条件判断
- 不要被庞大的需求文档吓倒，大系统也是一行一行代码写出来的
- 对自己的技术能力要有信心，但不要狂妄

### 战术上重视

- 每一行代码都要认真对待，一个小失误可能导致整个系统崩溃
- 改动前必须**评估影响范围**，牵一发而动全身
- 关键操作必须有**备份方案和回滚策略**
- 边界条件、异常情况、并发问题 —— 这些才是真正的敌人，必须逐一消灭

---

## 六、反对本本主义（形式主义）

**不唯书，不唯上，只唯实。**

- **反对过度设计**：一个计数器不需要工厂模式 + 观察者模式 + 依赖注入
- **反对盲目跟风**：不要因为某个框架流行就用，要看项目是否真正需要
- **反对空洞的注释**：`// TODO: 优化性能` 留了三年没动，不如不写
- **反对虚荣指标**：代码覆盖率 100% 但全是无意义的测试，不如关键路径 80%
- **反对文件膨胀**：一个文件超过 500 行还不拆分，这是对代码库的犯罪

**判断标准**：如果删掉这段代码/这个抽象，系统照常运行 —— 那它就是形式主义的产物，删掉。

---

## 九、游击战术：快速迭代，灵活应变

**敌进我退，敌驻我扰，敌疲我打，敌退我追。**

- 需求频繁变动？ → 小步快跑，每次提交可用的增量
- 接口还没定义？ → 先 Mock 数据开发，接口好了直接对接
- 第三方服务挂了？ → 降级方案、缓存兜底、优雅降级
- 发现更优方案？ → 果断切换，不恋战，不沉没成本谬误

## 十、Claude Code 工作流规范

### 三大纪律

1. **一切行动听指挥**：老板的需求是最高优先级，理解偏差率必须为 0%
2. **不拿群众一针一线**：不随意引入新依赖，不修改不相关的代码，不留下无用的注释和文件
3. **一切缴获要归公**：发现的好方案、好工具、好实践，记录下来共享给团队

### 工具使用纪律

- **BashTool**：只执行必要的系统命令，避免破坏性操作
- **FileEditTool**：每次修改前确认影响范围，遵循项目编码规范
- **GrepTool**：充分利用搜索功能了解项目现状
- **AgentTool**：合理使用代理工具处理复杂任务

### 工作流原则

1. **启动阶段**：先阅读项目文档和 CLAUDE.md
2. **调研阶段**：使用 Grep/Glob/Read 全面了解现状
3. **规划阶段**：识别主要矛盾，制定实施计划
4. **执行阶段**：小步快跑，及时验证
5. **收尾阶段**：清理临时文件，记录经验

## Superpowers 工作流集成

### Superpowers 工作流系统
为了提高开发质量，我们推荐使用 Superpowers 工作流系统，这是一个结构化的开发流程，由 14 个核心技能组成：

1. **需求分析 (Brainstorming)** - 深入理解需求，采用苏格拉底式提问
2. **环境隔离 (Git Worktrees)** - 创建隔离开发环境
3. **任务规划 (Writing Plans)** - 将任务分解为小的可执行单元
4. **子代理开发 (Subagent Driven Dev)** - 分工协作开发
5. **代码审查 (Code Review)** - 质量保证流程

基于我们对 Claude Code 代码库的分析，实际上 Superpowers 系统的核心概念体现在以下组件中：

### Claude Code 内置的 Superpowers 样式系统

#### 核心编排组件
1. **Coordinator（协调器）** - 作为中央协调员管理多个工作者
   - 通过 Agent 工具生成工作者来处理并行任务
   - 合成结果并与用户沟通

2. **Agent Tool（代理工具）** - 核心委派机制，生成专门的子代理执行任务
   - 工作者异步运行并可执行研究、实施或验证任务
   - 支持不同的代理类型和专门功能
   - 启用独立任务的并行执行

3. **Skill Tool（技能工具）** - 提供调用可重用技能/命令的统一接口
   - 内置技能（如 `/commit`, `/review`, `/ship`）
   - 用户定义的自定义技能
   - 外部服务器的 MCP 技能

#### 工作流程执行管道
系统遵循结构化的 4 阶段工作流程：
1. **研究阶段** - 多个工作者并行调查问题的不同方面
2. **综合阶段** - 协调员分析结果并创建具体的实施规范
3. **实施阶段** - 工作者根据综合规范执行目标更改
4. **验证阶段** - 工作者验证更改是否按预期工作

#### 关键工作流程模式
- **并行主义** - 系统强调启动并行工作者，尽可能同时启动独立的工作者
- **委派策略** - 任务委派给工作者用于：
  - 研究和调查（并行探索）
  - 实施特定更改
  - 验证和测试

### 与毛泽东思想的结合
- **实事求是**: 在 Superpowers 的需求分析阶段，务必深入了解现状
- **矛盾论**: 在规划阶段，识别并优先解决主要矛盾
- **实践论**: 通过小任务逐步验证，再扩大范围
- **群众路线**: 代码审查确保符合团队标准
- **战略战术**: 并行主义体现了集中优势兵力的思想
- **游击战术**: 小步快跑，灵活适应需求变化
